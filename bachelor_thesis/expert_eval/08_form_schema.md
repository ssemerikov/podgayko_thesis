# Схема Google Form / LimeSurvey для рубрики

Цей документ описує структуру і налаштування цифрової реалізації рубрики `06_rubric.md` у вигляді Google Form або LimeSurvey-опитування. Альтернативний шлях (заповнюваний PDF) — описано у кінці документа.

## Загальні налаштування форми

| Параметр | Значення |
|---|---|
| Назва форми | «Експертне оцінювання прототипу системи моніторингу П(ПТ)О» |
| Опис | (вступний текст з `06_rubric.md`, перші ~5 рядків) |
| Мова | Українська |
| Збір email | **Не збирати**, інакше зруйнується анонімність |
| Авторизація користувача | Без авторизації (доступ за прямим посиланням) |
| Обмеження на 1 заповнення | Так, по cookie (зменшує дублювання, але не блокує повторне заповнення) |
| Можливість редагування після відправки | Так (експерт може повернутися й уточнити) |
| Збір IP-адреси | **Ні** (відключити в LimeSurvey; в Google Form це автоматично) |
| Сторінка подяки | «Дякую! Ваші відповіді отримано. Якщо передумаєте чи зрозумієте, що залишилися думки — напишіть автору на email.» |
| Експорт відповідей | CSV / TSV; автоматичне надсилання на email автора щоденно |
| Notification | Email-сповіщення на serhi.pidhaykoy@gmail.com при кожному поданні |

## Структура секцій

### Секція 0: Вступ (інформаційна сторінка)

**Заголовок:** «Перед заповненням»

**Текст:**

> Ця форма займає 15–25 хв. Не поспішайте. Якщо щось забули — поверніться до карти сценаріїв або відеозапису. Усі Ваші відповіді знеособлюються до коду EXP-<n>; жодного зв'язку «ім'я ↔ код» не зберігається.
>
> На кінці є відкриті запитання — там зосередьтеся на 1–3 найважливіших спостереженнях.
>
> Якщо передумаєте, можете відкликати згоду в будь-який момент.

**Кнопка:** «Розпочати»

---

### Секція 1: Розділ A — System Usability Scale (10 пунктів)

**Тип питань:** Лінійна шкала 1–5 з підписами:
- 1 = Категорично не погоджуюсь
- 5 = Категорично погоджуюсь

**Питання A.1 — A.10** (з рубрики, дослівно).

**Налаштування:** усі 10 — обов'язкові.

---

### Секція 2: Розділ B — Доменно-специфічні пункти (8 пунктів + free-text)

Для кожного з 8 пунктів — комбінація:
1. Лінійна шкала 1–5 (обов'язкове).
2. Текстова відповідь «Коментар (опційно)» — до ~500 символів (опційне).

**Питання B.1 — B.8** (з рубрики, дослівно).

---

### Секція 3: Розділ C — Відкриті запитання (4 пункти)

| Пункт | Тип | Обов'язковість |
|---|---|---|
| C.1 — Що було найзручнішим? | Текстова відповідь, до ~1000 символів | Обов'язкове |
| C.2 — Що викликало найбільші труднощі? | Текстова відповідь, до ~1000 символів | Обов'язкове |
| C.3 — Що бракує для впровадження? | Текстова відповідь, до ~1000 символів | Обов'язкове |
| C.4 — Self-assessment TRL | Випадаючий список 1–9 + текстове обґрунтування | Обов'язкове |

---

### Секція 4: Розділ D — Метадані (6 пунктів)

| Пункт | Тип | Обов'язковість |
|---|---|---|
| D.1 — Поточна позиція | Випадаючий список (7 опцій з рубрики) + «Інша» з текстом | Обов'язкове |
| D.2 — Регіон | Випадаючий список (6 опцій з рубрики) + «Інше» з текстом | Обов'язкове |
| D.3 — Стаж | Радіо-кнопки (5 опцій) | Обов'язкове |
| D.4 — Переміщений заклад | Радіо-кнопки (4 опції) | Обов'язкове |
| D.5 — Технологічний рівень | Радіо-кнопки (4 опції) | Обов'язкове |
| D.6 — Згода на half-day | Радіо-кнопки (3 опції) | Обов'язкове |

---

### Секція 5: Підпис

| Пункт | Тип | Обов'язковість |
|---|---|---|
| Дата заповнення | Дата | Автоматично |
| Тривалість заповнення (хв) | Число | Опційне |
| Технічні проблеми? | Радіо-кнопки (Так/Ні) + текст | Опційне |

**Кінцева сторінка:** «Дякую за час та внесок. Анонімізовані результати з'являться у дисертації.»

## Експорт даних

### Налаштування експорту з Google Form

1. У Sheets-таблиці, прив'язаної до форми, виставити кодування UTF-8.
2. Експортувати у CSV з роздільником `,` (англомовний коми).
3. Назва файлу: `raw_responses_EXP-<n>.csv` (по одному файлу на експерта; це робиться вручну: завантажуємо повний CSV, фільтруємо за timestamp, вирізаємо рядок експерта).
4. Кожен `raw_responses_EXP-<n>.csv` має один рядок-header + один рядок-data.
5. Після експорту — видалити рядок з Sheets (бо містить його в plain text).

### Заголовки CSV (стандартизовано)

```csv
respondent_code,timestamp,
sus_a1,sus_a2,sus_a3,sus_a4,sus_a5,sus_a6,sus_a7,sus_a8,sus_a9,sus_a10,
b1_score,b1_comment,b2_score,b2_comment,b3_score,b3_comment,b4_score,b4_comment,
b5_score,b5_comment,b6_score,b6_comment,b7_score,b7_comment,b8_score,b8_comment,
c1_text,c2_text,c3_text,c4_trl,c4_justification,
d1_role,d1_other,d2_region,d2_other,d3_experience,d4_displaced,d5_tech_level,d6_halfday,
duration_min,tech_issues,tech_issues_text
```

(Ця схема вживається у `eval_aggregate_experts.py` і в `responses_template.csv`.)

## Заповнюваний PDF (fallback)

Якщо експерт надає перевагу offline-форматі:

1. Згенерувати з `06_rubric.md`:
   ```bash
   pandoc 06_rubric.md -o 06_rubric_fillable.pdf \
       --template=eisvogel \
       -V geometry:a4paper \
       -V fontsize:11pt \
       --pdf-engine=xelatex \
       -V mainfont="Times New Roman" \
       -V lang=uk
   ```
2. Через PDF-редактор (Adobe Acrobat або PDFescape) додати form-fields:
   - Радіо-групи для шкал 1–5 (сума 10×5 + 8×5 = 90 полів).
   - Текстові поля (multi-line) для коментарів і відкритих запитань.
3. Зберегти як `06_rubric_fillable.pdf`.
4. Експерт повертає заповнений PDF → автор парсить через `eval_pdf_to_csv.py` → отримує `raw_responses_EXP-<n>.csv` у тій самій схемі.
